5. **Quick Switcher** (`Ctrl/Cmd + O`) for instant navigation to any topic

### For Learning

> [!tip] Recommended Learning Path
> If you are new to architecture, begin with the **Principles** folder to understand the "why" behind code structure.

1. **Start with the Basics:** [[SoftwareArchitecture/Principles/SOLID|SOLID]], [[SoftwareArchitecture/Principles/DRY_KISS_YAGNI|DRY/KISS/YAGNI]] to understand the "why" behind code structure.
2. **Explore Patterns:** Move to [[SoftwareArchitecture/DesignPatterns/|Design Patterns]] to see standard solutions to common problems.
3. **Scale Up:** Study [[SoftwareArchitecture/ModernArchitectures/|Modern Architectures]] and [[SoftwareArchitecture/BestPractices/|Best Practices]] to learn how to build production-ready systems.
4. **Specialize:** Dive into [[SoftwareArchitecture/DataAndAI/|Data & AI]] or [[SoftwareArchitecture/Cybersecurity/|Cybersecurity]] based on your interest or project needs.

### For Reference

Each Markdown file is self-contained. You can quickly look up a concept (e.g., "Circuit Breaker" or "RAG Pattern") to find:
- A clear explanation
- A code snippet (Python/JS)
- A diagram or flow description
- Pros and cons

### Visuals and Code
- **Diagrams:** We use ASCII and Mermaid diagrams to visualize complex flows.
- **Code:** Examples are provided in Python and JavaScript to be accessible to most developers.

## ü§ù Contributing

We believe that knowledge grows when shared. We welcome contributions from the community!

1. **Fork** the repository.
2. **Branch** off from `main` (`git checkout -b feature/new-pattern`).
3. **Add** your content (Markdown files, diagrams, examples).
4. **Commit** your changes with a descriptive message.
5. **Push** to your fork and submit a **Pull Request**.

**Ideas for contributions:**
- Add a new design pattern.
- Provide an example in a different language (Go, Rust, Java).
- Improve existing diagrams.
- Write a guide on a new trend (e.g., Quantum Computing).
- Enhance Obsidian links and connections.
